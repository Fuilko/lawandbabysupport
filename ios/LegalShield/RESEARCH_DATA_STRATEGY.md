# LegalShield 研究數據追蹤策略
## 如何在鐵壁隱私下做有價值的研究

---

## 一、核心矛盾

> **研究需要數據 → 個資法禁止收集敏感個資 → 沒有 IRB + 明確同意，不能碰第一手案件內容**

所以我們設計了三條獨立且互補的數據管道：

```
┌─────────────────────────────────────────────────────────────────┐
│                    三軌並行研究數據策略                           │
├─────────────────┬─────────────────┬─────────────────────────────┤
│   Track A       │   Track B       │   Track C                   │
│   端內匿名統計   │   平台去識別化   │   合作夥伴研究協議           │
│   [PHONE]       │   [BACKEND]     │   [PARTNER + UNIVERSITY]    │
├─────────────────┼─────────────────┼─────────────────────────────┤
│ 差分隱私 ε=1.0  │ 案件處理流程數據 │ 匿名化後的案件統計            │
│ 本地計算        │ 不含個資內容     │ 大學/研究機構正式合作          │
│ 用戶可控上傳    │ 後端自動生成     │ IRB 審查 + 倫理委員會         │
└─────────────────┴─────────────────┴─────────────────────────────┘
         │                │                      │
         ▼                ▼                      ▼
    ┌─────────┐     ┌─────────┐            ┌─────────┐
    │ 產品迭代 │     │ 平台優化 │            │ 學術論文 │
    │ 改善模型 │     │ 路由調校 │            │ 補助申請 │
    └─────────┘     └─────────┘            └─────────┘
```

## 二、Track A：端內匿名統計（PHONE）

### 設計原則
- **原始數據不出裝置**：案件內容、證據、GPS 精確座標永遠留在 iPhone
- **本地計算**：統計在端內完成，後端只收到「噪聲後的數字」
- **用戶可控**：預設關閉，需主動開啟且可隨時關閉

### 收集什麼（僅結構化數字）

```swift
// iPhone 本地累積的原始計數（不上傳）
struct LocalRawCounter {
    var childAbuseCases: Int      // 兒少保護案件數
    var laborCases: Int           // 勞動案件數
    var contractScans: Int       // 契約掃描次數
    var avgEvidencePerCase: Double // 平均每案證據數
    var leadingQuestionDetections: Int // 誘導問句偵測次數
    var emergencyTriggers: Int    // 緊急按鈕觸發次數
    var partnerResponseTimeAvg: TimeInterval // 合作夥伴平均回應時間
}
```

### 差分隱私處理（上傳前）

```swift
// Laplace 機制：每個計數加入 Laplace(0, 1/ε) 雜訊
// ε = 1.0（強隱私級別）
func addLaplaceNoise(count: Int, epsilon: Double = 1.0) -> Int {
    let scale = 1.0 / epsilon
    let noise = randomLaplace(scale: scale)
    return max(0, count + Int(round(noise))) // 截斷負數
}

// 範例：
// 真實值：childAbuseCases = 3
// 加入 noise = -1.7
// 上傳值：1（無法反推「這個用戶有 3 個兒少案件」）
```

### 上傳的資料包範例

```json
{
  "uploadId": "uuid",
  "weekStarting": "2024-05-13",
  "epsilon": 1.0,
  "deviceCountry": "JP",
  "categoryDistribution": {
    "child_abuse": 1,      // ← 真實值 3，加入 noise 後變 1
    "labor_exploitation": 0, // ← 真實值 1，加入 noise 後變 0
    "contract_trap": 2       // ← 真實值 2，加入 noise 後變 2
  },
  "interviewStats": {
    "avgLeadingQuestions": 2.3,  // ← 平均值雜訊影響較小
    "avgDurationMinutes": 45.0
  },
  "evidenceTypeDistribution": {
    "photo": 5,
    "audio": 2,
    "document": 1
  }
}
```

### 誰也不能反推個人

- 攻擊者知道：「這個用戶上傳了 child_abuse: 1」
- 攻擊者無法知道：這是「真實 3 + noise -2」還是「真實 1 + noise 0」
- 攻擊者無法知道：案件內容、當事人姓名、GPS 座標

## 三、Track B：平台去識別化數據（BACKEND）

### 設計原則
- **後端從不儲存原始個資**：收到資料後立即去識別化
- **只保留「流程數據」**：處理時間、路由結果、狀態轉換
- **自動生成**：研究統計由後端自動計算，不需要人工接觸個案

### 收集什麼（平台營運數據）

| 數據項 | 範例 | 說明 |
|--------|------|------|
| 案件路由時間 | 從觸發到分派平均 8 分鐘 | 不含案件內容 |
| 合作夥伴回應率 | 90% 在 10 分鐘內確認 | 匿名夥伴 ID |
| 去識別化案件類型分布 | 兒少 40% / 勞動 30% / 消費者 20% | 僅類型 |
| 行政救濟期限逾期率 | 15% 的案件錯過審査請求期限 | 引導產品改善 |
| 契約掃描高風險條款 Top 10 | 「放棄一切權利」佔 35% | 純文字模式 |
| 法律問答熱門主題 | 「過労死ライン」佔 20% | 引導 RAG 擴充 |

### 關鍵：後端「天生」不知道個資

```
📱 iPhone 上傳 → {hash陣列, 區域, 類型, 時間戳}
                    ↓
🖥️ 後端收到 → 立刻分類儲存（按類型，不按個人）
                    ↓
後端只知道：「東京都新宿區，今天有 3 件兒少案件被觸發」
後端不知道：「這 3 件是誰、什麼內容、證據在哪」
```

### 平台數據的用途

1. **產品迭代**：「勞動案件平均回應時間 15 分鐘，太慢了，需要優化路由」
2. **合作夥伴管理**：「A 夥伴回應率 95%，B 夥伴 60%，需要跟進 B」
3. **補助申請佐證**：「平台運作 3 個月，處理 127 件案件，平均回應 8 分鐘」

## 四、Track C：合作夥伴研究協議（PARTNER + UNIVERSITY）

### 為什麼需要這條軌道

Track A 和 B 只能回答「多少」「多快」「什麼類型」。
要回答「為什麼」「怎麼改善」「法律政策建議」，需要更深入的質性/量性研究。
但這必須通過**正式的學術倫理審查**。

### 運作方式

```
LegalShield 平台 ──去識別化統計──▶ 大學研究團隊
                                    │
                                    ▼
                    ┌──────────────────────────┐
                    │   大学 IRB（研究倫理委員会） │
                    │   審查通過後，才能：          │
                    │   • 訪談合作夥伴專員           │
                    │   • 分析匿名化案件檔案         │
                    │   • 發表論文                  │
                    └──────────────────────────┘
                                    │
                                    ▼
                    ┌──────────────────────────┐
                    │  研究產出：                │
                    │  • 論文（Digital Forensics）│
                    │  • 政策建議書               │
                    │  • 教材（專員培訓用）        │
                    │  • 補助申請背書             │
                    └──────────────────────────┘
```

### 研究數據來源（經 IRB 審查後）

| 來源 | 內容 | 取得方式 |
|------|------|----------|
| 合作夥伴案件統計 | 去識別化後的年度報告（案件數、類型、結案率） | 夥伴年度報告中已公開的統計 |
| 專員訪談 | 5-10 位專員的深度訪談（「您覺得平台哪個功能最有用？」） | 研究團隊執行，IRB 同意後錄音 |
| 匿名化案件時間軸 | 「案件 A：從觸發到結案歷時 45 天，經歷 3 個狀態轉換」 | 後端自動生成，移除所有個資 |
| 法律問答日誌 | 熱門問題關鍵詞（「過勴死」佔 20%） | 後端關鍵詞統計，不含用戶 ID |

### IRB 審查的關鍵問題

IRB（研究倫理委員會）會問：

1. **「你們怎麼確保無法識別個人？」**
   → 回答：三軌分離。Track A 有 DP noise，Track B 無個資，Track C 經 IRB 審查後才訪談。

2. **「用戶有沒有明確同意參與研究？」**
   → 回答：緊急轉介同意書中有「研究同意」獨立勾選項，預設關閉，可隨時撤銷。

3. **「研究結果會不會傷害弱勢群體？」**
   → 回答：所有發表論文需經合作夥伴（児童相談所）審閱，確保不暴露系統漏洞。

4. **「數據儲存在哪？安全嗎？」**
   → 回答：原始數據在 iPhone（用戶控制），後端統計在 AWS Tokyo 區域（加密），研究數據在大學安全伺服器。

## 五、研究閉環：數據如何回流到產品

```
Step 1: 產品使用（Track A/B 自動收集）
    ↓
Step 2: 每週/每月產出統計報告
    ↓
Step 3: 研究團隊分析（Track C 深度研究）
    • 發現：「85% 的兒少案件在首次通報時缺少證據照片」
    • 建議：「App 應在通報流程中主動提示『請拍攝傷口照片』」
    ↓
Step 4: 產品迭代
    • 更新 App 流程：通報頁面新增「拍照提示」
    • 更新 LLM prompt：強化證據收集建議
    ↓
Step 5: 效果驗證（下一週期統計）
    • 「拍照提示」上線後，兒少案件證據率從 15% 提升到 62%
    ↓
Step 6: 論文發表 + 補助申請
    • 論文：「Digital Nudge in Child Abuse Reporting: Evidence from a Mobile Platform」
    • 補助：用這篇論文申請 KAKENHI 基盤研究
```

## 六、三軌數據的具體產出與用途

| 數據軌道 | 產出頻率 | 誰使用 | 用途 |
|----------|----------|--------|------|
| **Track A** 端內 DP 統計 | 每週批次 | 產品團隊 | A/B 測試、功能優先級調整 |
| **Track B** 平台營運數據 | 即時/日報 | 平台營運 | 夥伴管理、路由調校、容量規劃 |
| **Track C** 研究協議 | 每季/每年 | 學術團隊 | 論文、政策建議、補助申請、專員培訓教材 |

## 七、行動清單（研究數據面）

### 立即（Phase 0）
- [ ] 在「緊急轉介同意書」中加入「研究同意」獨立勾選項
- [ ] 確保 ResearchDataManager（差分隱私模組）預設為「關閉」，需用戶主動開啟

### Phase 1（0-3 月）
- [ ] 完成 Track A：iPhone 本地 DP 統計計算 + 測試 noise 分佈
- [ ] 完成 Track B：後端平台儀表板（僅營運數據，無個資）
- [ ] 聯繫日本大學法務研究科，確認 IRB 合作意願

### Phase 2（3-6 月）
- [ ] 第一個合作夥伴導入後，啟動 Track B 數據收集
- [ ] 提交 IRB 申請（以「平台使用經驗調查」為主題，非「案件內容分析」）
- [ ] 開始撰寫第一篇白皮書：「児童虐待通報デジタル化の現場検証」

### Phase 3（6-12 月）
- [ ] 發表第一篇論文（Digital Investigation 或實務期刊）
- [ ] 用論文 + 平台數據申請 KAKENHI
- [ ] 建立「研究 Dashboard v2」：視覺化三軌數據

---

> **核心原則：研究價值不在於「知道誰發生了什麼」，而在於「知道多少、多快、怎麼改善」。**
>
> **個資是負債，統計才是資產。**
